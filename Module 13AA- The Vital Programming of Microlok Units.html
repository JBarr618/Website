Module 13A: The Vital Programming of Microlok Units

The vital programming of Microlok units is typically done using proprietary software and a specific, high-level language developed by the manufacturer (originally GRS, now Wabtec).1 This language is designed with safety as the paramount concern, ensuring that the interlocking logic implemented is robust and prevents unsafe conditions.

[Image: Screenshot of Microlok IDE Interface]

Here’s what we know about the software and language used:

  • Proprietary Nature: The exact details of the language and development environment are generally proprietary to Wabtec. This is common in safety-critical systems to maintain security and control over the development and verification processes.
  • High-Level Language: The programming language is a high-level language that allows signaling engineers to define the interlocking logic in a way that is more intuitive and less error-prone than working directly with machine code or low-level languages. It uses constructs and terminology familiar to signaling principles (e.g., conditions for route locking, signal aspects based on track occupancy and switch positions).
  • Focus on Safety: The language and the associated compiler are designed with built-in checks and constraints to help ensure the safety integrity of the programmed logic.2 This includes features for defining vital outputs and ensuring that they are only activated under safe conditions.
  • Development Environment: Wabtec provides a specific software development environment that includes tools for writing the interlocking logic in the proprietary language, compiling it into a format that can be executed by the Microlok unit, simulating and testing the logic offline, and downloading it to the Microlok hardware.
  • Documentation: Comprehensive documentation is provided by Wabtec to guide signaling engineers in using the programming language and development tools effectively and safely. This documentation often includes rules and guidelines for writing vital logic.
[Image: Microlok Workflow Diagram]

While the specific name and syntax of the Microlok vital programming language are proprietary, the general principles involve a high-level, safety-focused language used within a dedicated development environment provided by the manufacturer. This approach ensures that the complex interlocking logic required for safe railway operation can be implemented and verified with a high degree of confidence.

It’s worth noting that other EPTC systems from different manufacturers (like Siemens’ IVPI or various Electrologixs implementations) will have their own proprietary vital programming languages and development environments tailored to their specific hardware and safety architectures.
